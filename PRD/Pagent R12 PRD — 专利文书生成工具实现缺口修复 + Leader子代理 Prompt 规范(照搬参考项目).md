## 2. 现状核对结论(工具实现缺口)

四个工具文件都存在且已在 `build_default_tool_registry` 注册,但"注册 ≠ 真正实现":

### 2.1 子代理工具 — 空壳(未实现)

`app/tools/subagents/__init__.py` 定义了 8 个 `SubagentDefinition` 和一个通用 `PatentDraftingSubagentTool`,但 `run()` **完全没有调用 LLM**:只是读 `source_artifact_key` → 前面拼 `# 标题` → 写回 `output_artifact_key`。

- 无 role prompt、无 SOP、无按大纲生成、无工具访问。
- 结论:骨架在,生成能力为零,等于未实现。

### 2.2 patent_search — 占位 stub(未实现)

`patent_search.py` 自称"占位工具":空 query 报错;`allow_network=False` 返回 `network_disabled`;否则返回假 evidence `patent_search skipped: {query}` 且 `sufficient=False`。从不真正检索。

### 2.3 draft_workspace — 已实现但偏离(缺工作区维持能力,本轮新增)

`draft_workspace.py` 的读写**真的能用**(磁盘落盘 + key 白名单校验 + 截断 + 防路径逃逸),但缺最基本的"维持工作区"能力:

- **没有目录/项目工作区概念**:参考项目靠 `temp_[uuid]/01_input … 05_final` 目录结构在子代理间流转数据,当前只有扁平的 `{key}.md`,无法承载分阶段目录。
- **没有 `list` 动作**:leader 无法枚举已产出 artifact,做不了"任务审查/缺文件重委托"。
- **没有合并能力**:description part2 与 markdown_merger 都依赖"文件合并",当前无对应动作。
- **磁盘恒定**:跳过了 R11 定的"内存 key-store 为默认、磁盘为可选降级"。

### 2.4 skill_loader — 概念理解错误

`skill_loader.py` 的 `_ALLOWED_SKILLS` 把 skill 名映射到 `.py` 文件并 `read_text()`,即**把可执行 skill 类的 Python 源码当技能内容返回**。双重误解:

- 误解 Pagent 原有 skill:`app/skills/*.py`(如 `PatentQASkill.run(context: SkillContext)`)是消费 `SkillContext` 产出结构化结果的**可执行能力类**,不是文档;读其源码对撰写毫无价值,还把实现细节倒进数据层,破坏指令/数据分离。
- 误解参考项目 `learn_skills`:它读的是 **Markdown 技能/SOP 文档**(领域 know-how,对应 patent-disclosure-skill 那种 [SKILL.md](http://SKILL.md)),不是代码。

## 3. 修复要求

### 3.1 draft_workspace 增强(工作区能力)

- 引入**项目工作区**:以 `temp_[uuid]/` 为根,内建 `01_input / 02_research / 03_outline / 04_content / 05_final` 目录结构;artifact 用"相对路径 key"寻址(如 `04_content/abstract`)。
- 新增动作:`list`(枚举目录/artifact,供 leader 审查)、`merge`(按顺序合并多个 artifact 为一个)。
- 默认内存 key-store,`PAGENT_DRAFT_WORKSPACE_DIR` 非空时落盘;保留 key 白名单 + 防逃逸。
- 保留现有 read/write 语义:长正文只走工作区,入参只传 key/路径。

### 3.2 子代理工具 → 真正的 subagent

- 每个子代理 = LLM + 各自 role SOP prompt(见 §4)+ 受限工具访问(draft_workspace / skill_loader / office_to_md·file_extract / patent_search 按角色分配)。
- 输入只传 `source_artifact_key`/路径,产出写回工作区并返回 `{artifact_key, done}`,不回传长正文。

### 3.3 patent_search → 真实检索

- 接入真实检索后端(对齐参考项目 google_patent_search 用途:按关键词检索、优先 CN、GRANT、Top-K),受联网门控,离线安全降级。

### 3.4 skill_loader → 读 Markdown 技能文档

- 从**独立技能文档目录**(`PAGENT_SKILL_DIR`,如 `app/skills_docs/*.md`)按白名单读取 Markdown 领域知识/SOP,与 `app/skills/*.py` 可执行模块**彻底解耦**;命名上区分"可执行 Skill 类"与"可加载技能文档"。

### 3.5 TodoList 规划中间件 / `write_todos` 工具(新增,必须实现)

- 背景:§4.2 的 todo prompt 本身不干活,真正的 `write_todos` 工具、todo 列表状态与上下文注入,在参考项目中由 LangChain `TodoListMiddleware` 提供(leader 与全部 9 个子代理都挂载)。Pagent 为自研 bounded ReAct,无等价内置,必须自行实现,否则 §4.2 的 prompt 形同空转。
- 实现要点:
  - 提供 `write_todos` 工具:入参为完整 todo 列表(每项含 text / status:pending·in_progress·done)。
  - 在 agent 状态中维护该 todo 列表,每步把当前列表注入 leader / 子代理上下文。
  - 使用 §4.2 的 sys_prompt 作为规划说明、tool_prompt 作为工具描述。
  - 与 R11 的步数/预算门控协同(`PAGENT_DRAFTING_MAX_STEPS` 等);todo 状态本身不改变硬门控。
- 挂载范围:对齐参考项目——leader 与每个子代理各自挂一份。

### 3.6 `parsed_info` 结构(可选增强,默认保持软约定)

- 现状:参考项目 `parsed_info.json` 无代码级 schema,结构仅由 `input_parser` prompt 的字段清单约定,下游子代理把它当文本读;好处是零耦合,代价是字段缺失/格式错不会被发现,只靠 leader 的任务审查(看文件是否存在)兜底。
- 默认:保持与参考一致的软约定(符合本 PRD 照搬、减少结构化的方向)。
- 可选增强(如需更强可靠性):为 `parsed_info` 定义 pydantic / JSON Schema,让 `input_parser` 走结构化输出 + 校验失败重试。此项与减少结构化的取向有张力,列为可选、不默认启用。

在以下情况下，应避免使用此工具：
只有一个简单明了的任务
2. 这项任务微不足道，追踪它没有任何益处
3. 这一任务仅需不到3个简单步骤即可完成
4. 该任务纯属对话或信息类任务

1. 任务状态：使用这些状态来跟踪进度：
   - pending: 任务尚未开始
   - in_progress：当前正在进行中（如果多个任务之间没有关联且可以并行运行，则你可以同时有多个任务处于in_progress状态）

   - 在工作中实时更新任务状态
   - 完成任务后立即标记为完成（不要批量完成）
   - 在开始新任务之前，先完成当前的任务

3. 任务完成要求：
   - 只有在任务完全完成时，才能将其标记为已完成
   - 如果遇到错误、阻碍或无法完成，请将任务状态保持为“进行中”
