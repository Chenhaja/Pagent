# 专利 Agent 架构设计 — 编排 / Node / Tool / Skill 与 ReAct 融合

<aside>
🎯

**总体判断**：外层用确定性编排（DAG / 状态机），内层只在少数「开放式」节点里局部使用 bounded ReAct。不要把整个专利 agent 做成一个大 ReAct loop，也不要完全不用 ReAct。卡点的根因是把「编排范式」和「执行策略」压到了同一层——分开即解。

</aside>

## 一、核心心智模型

> **编排连的是 node；node 执行时装载一个 skill（怎么做）+ 调用若干 tool（用什么做）+ 读写全局 state。**
> 

口诀：

- 「流程的一步」 → **node**
- 「一种原子能力」 → **tool**
- 「怎么写得专业」 → **skill**
- 「这一步要不要边想边做」 → 决定该 node **用不用 ReAct**

---

## 二、Tool / Node / Skill 的边界

| 维度 | Tool（工具） | Node（节点） | Skill（技能） |
| --- | --- | --- | --- |
| 本质 | 原子能力 | 业务流程的一步 | 把某件专业事做好的方法包 |
| 粒度 | 最小、无业务语义 | 一次有意义的状态推进 | 一类专业写作 / 分析能力 |
| 有无状态 | 无（stateless） | 读写全局 state | 无（被加载时注入上下文） |
| 是否决策 | 否，纯执行 | 是，决定本步怎么走 | 否，是知识 + 模板 |
| 构成 | 一个函数 / 一次 API | 加载 skill + 调若干 tool + 局部推理 | prompt 模板 + few-shot + 领域规则 + 输出 schema |
| 例子 | 专利检索 API、向量召回、引用关系校验器、术语库查询 | 技术特征抽取、独权撰写、合规自检 | 独权撰写法、从权层级拆分法、新颖性比对法 |

---

## 三、编排的最小单位 = node，不是 tool

编排层真正关心的是**可观测、可重试、可回滚、可分支、可人审**——这些天然以「业务步骤」为粒度。

- 用 tool 当最小单位 → 编排图爆炸，且耦合实现细节（换检索 API 就要改图）。
- 用 skill 当最小单位 → 它是「方法」不是「流程步骤」，没法连边。

### 判定：该是 node 还是退化成 tool / 内部步骤

满足越多越应该独立成 node：

- [ ]  它让 state 发生一次**能被命名**的推进（交底书 → 结构化特征清单）
- [ ]  它的输出能被一个**明确的检查**验证（rule 或 LLM-judge）→ 才能独立重试
- [ ]  它是**失败隔离点**（依赖外部 API、容易失败）
- [ ]  它是**人机交互边界**（需要人审 / 介入）
- [ ]  它**可缓存 / 可复用**

**反例**（应是 tool 或 node 内步骤）：纯格式化、字符串拼接、单一性检查、一次无独立语义的 LLM 调用。

## 四、整体分层架构

```
接入层（前端 / API 网关：鉴权、限流）
   │
轻量改写（指代消解 / 上下文补全 / 降噪，意图无关）
   │
意图识别 & 路由层（撰写权利要求 / 现有技术检索 / OA答复 / 全流程代写 → 路由到预定义 workflow）
   │
（意图相关的深度改写：按 workflow 需要的 schema 规范化，可选）
   │
编排层 Orchestrator（管理 workflow DAG / 状态机：调度 node、重试、分支、并行、human-in-loop、预算控制）
   │
执行层 Nodes（业务步骤，每个 node 选择执行模式）
   │
能力层（Skills 方法包 + Tools 原子能力 + 知识库：专利法/审查指南/撰写规范/术语库/模板库，RAG）
   │
模型层（Embedding / Reranker / 生成 LLM，敏感走本地）+ 数据合规层（不出域）
```

<aside>
⚠️

意图识别后不是临时拼装散任务，而是**路由到预定义 workflow 模板**。专利流程强规范、强可审计，骨架必须确定性，不能让模型自由决定步骤顺序。

</aside>

## 五、改写层：注意有两种「改写」，别混

|  | 输入改写（对话层） | 检索 query 改写（N2 内部） |
| --- | --- | --- |
| 位置 | 意图识别**之前** | 检索 node **内部** |
| 目标 | 把这轮话变成独立、完整、规范的请求 | 把请求扩成多个适合召回的检索式 |
| 典型操作 | 指代消解、上下文补全、术语规范化、降噪 | 同义扩展、子查询拆分、HyDE、多路 query |
| 服务对象 | 意图识别 + 路由 | 向量 / 全文召回 |

### 输入改写该做什么

- **指代消解 + 上下文补全**：多轮里「再帮我把从权也写了」必须结合历史与 state 补全。
- **术语规范化**：口语 → 专利术语（「发明点」→ 技术特征），命中术语库。
- **噪声清理 + 复合意图标注**：去寒暄、错别字；一句话含多诉求则标注，交意图识别去拆。
- **不做**：不改变原意、不臆测补技术内容、**永远保留原文**（raw + 改写都进 state）。

### 设计张力：改写 vs 意图，谁先谁后

深度改写常需先知意图，而意图识别又想要干净输入。拆成两段：

```
轻量归一化（意图无关：消解 / 补全 / 降噪）
        │
    意图识别 & 路由
        │
意图相关的深度改写（按 workflow schema 规范化，可选）
        │
      编排层
```

工程上：前置改写是**轻量预处理步骤**（一次 LLM 或规则，无外部依赖、无回环），不做成重型 node，但输出要进 state（`raw_input` + `normalized_input` + `dialog_context`）。置信度低 / 改动过大时回退原文或触发澄清反问。

## 六、「权利要求书生成」任务的内部拆解

一个 workflow，节点间确定性连边 + 校验回环。这直接回答「执行节点 vs skill」：**node 负责流程位置与状态推进，skill 负责这一步怎么写得专业**。

| Node | 职责（state 推进） | 装载 Skill | 调用 Tool |
| --- | --- | --- | --- |
| N1 技术特征抽取 | 交底书 → 结构化技术特征 / 问题 / 效果，区分必要 vs 附加 | 特征抽取法 | LLM、术语库查询 |
| N2 检索 & 对比文件分析 | 找现有技术，定位区别特征，确定保护范围 | 新颖性 / 创造性比对法 | 检索 API、向量召回、Reranker、特征对照表生成器 |
| N3 权利要求布局规划 | 独权数量、独 / 从层级、产品 / 方法 / 系统 / 介质多类型布局 | 权利要求布局法 | LLM |
| N4 独立权利要求撰写 | 生成各独权 | **独权撰写 skill** | LLM、模板库 |
| N5 从属权利要求生成 | 逐层生成从权、维护引用关系 | **从权层级拆分 skill** | LLM、引用关系生成器 |
| N6 合规自检 | 单一性、引用正确、缺乏必要技术特征、说明书支持、清楚简要 | 审查规则集 | 规则校验器 + LLM-judge |
| N7 人审 / 修订 | 律师 / 发明人 review | — | UI 表单 |

**回环**：N6 / N7 不通过 → 带反馈回 N4 / N5 重写。

## 七、ReAct 怎么融合

<aside>
🧠

**核心**：ReAct 不是整个系统的编排范式，而是单个 node 内部可选的一种执行模式。把 ReAct 关进 node 这个笼子里。

</aside>

编排层给 node 提供三种执行模式：

| 模式 | 适用 | 对应 node |
| --- | --- | --- |
| (a) 确定性 / 链式 LLM | 步骤可预先规划 | N1 特征抽取、N3 布局、N5 从权 |
| (b) ReAct / agentic loop | 问题开放、步骤无法预先规划、要边想边调工具 | N2 检索与对比分析、创造性论证、OA 答复 |
| (c) 纯 tool / rule | 硬规则 | N6 的规则校验 |

**为什么不全 ReAct**：专利强规范、强可审计、合规敏感。外层若交给模型自由 ReAct，会不可复现、不可审计、保护范围出错风险高。但 N2 这类检索分析确实无法预先写死步骤，必须 ReAct。

### 内层 bounded ReAct 的护栏

- 限定工具集（该 node 只能用相关 tool）
- 最大步数 / token 预算 / 超时
- 结构化输出 schema（loop 结束必须产出可被 N6 校验的结构）
- 全程 trace 落库，便于审计

> 结论：**有必要融合，但要受限、局部**——外层确定性编排保证可控可审计，内层 bounded ReAct 解决开放式子问题。
>

## 八、落地清单

- [ ]  链路：轻量改写 → 意图识别 → 路由 workflow →（意图相关深度改写）→ 编排 nodes
- [ ]  编排连 node，node 内装 skill、调 tool，三层不要压扁
- [ ]  设计全局 state / blackboard：交底书、特征清单、检索结果、各级权利要求、校验报告、人审意见
- [ ]  N2 内保留检索 query 改写（与输入改写分开）
- [ ]  为每个 node 标注执行模式 (a)/(b)/(c) 与重试 / 回环策略
- [ ]  关键节点（权利要求）强制 human-in-the-loop
- [ ]  设计四类记忆（工作 / 会话 / 案件 / 长期）与写入门控
- [ ]  引入 LLM Wiki 作为长期记忆 / 画像的结构化表示（双层：原始日志 + 可编辑 wiki）

## 九、记忆体系（Memory）

<aside>
🧠

**核心**：记忆不是一个模块，而是按「时效 + 作用域」分层的四类。它们大多已藏在前面的架构里——工作记忆 = state，语义记忆 = 知识库 RAG，程序记忆 = skills；真正需要新增的是**会话记忆**与**案件记忆**。

</aside>

### 1. 先和已有结构对齐

| 记忆类型（认知分类） | 对应本架构的什么 | 是否需新建 |
| --- | --- | --- |
| 工作记忆 Working | 第四节的全局 state / blackboard | 已有 |
| 语义记忆 Semantic | 能力层知识库（专利法 / 审查指南 / 术语库 / 模板，RAG） | 已有 |
| 程序记忆 Procedural | Skills（独权撰写法等方法包） | 已有 |
| 会话记忆 Session | 支撑「轻量改写 / 指代消解」的多轮上下文 | **需新建** |
| 情景 / 案件记忆 Episodic | 跨会话持久化的案件历史与经验 | **需新建** |

### 2. 四类记忆的职责

| 类型 | 装什么 | 时效 / 作用域 | 存储 | 服务谁 |
| --- | --- | --- | --- | --- |
| 工作记忆 | 交底书、特征清单、各级权利要求、校验报告、人审意见 | 单次任务内，结束即归档或丢弃 | 内存 / state object | 所有 node |
| 会话记忆 | 多轮对话、用户这次的临时偏好与指代对象 | 会话级，滑窗 + 摘要压缩 | 会话缓存 + 摘要 | 改写层、意图识别 |
| 案件记忆 | 某专利案件全历史：交底书、各版权利要求、检索结果、OA 历史 | 长期、按 client / case 隔离 | 结构化 DB（版本 + 元数据）+ 向量 | N1 / N2 / N4 / N7 |
| 长期经验记忆 | 成功 / 被驳回模式、审查员偏好、用户撰写风格画像、可复用范式 | 长期、跨案件沉淀 | 向量库 + 画像表 | N2 / N3 / N4 |

### 3. 专利场景的特有难点

- **案件隔离与保密**：不同客户案件严格隔离，按 client / case 多租户；机密不出域（与合规层一致）。
- **版本化记忆**：权利要求多版本，必须可追溯「哪一版、为什么改、谁审的」——案件记忆要存版本链而非覆盖。
- **写入门控（memory gating）**：不是所有产出都进长期记忆，否则幻觉 / 噪声污染。**只有经过 N6 校验或 N7 人审通过的内容才固化进案件 / 经验记忆**。
- **冲突消解**：新信息与旧记忆矛盾时，标注来源与时间，优先可信来源（人审 > 校验 > 模型生成）。
- **遗忘 / TTL**：会话记忆过期清理；敏感信息按合规保留期限处理。

### 4. 读：在哪个 node 注入哪种记忆

| Node | 注入的记忆 |
| --- | --- |
| 改写层 | 会话记忆（指代消解 / 上下文补全） |
| N1 特征抽取 | 案件记忆（同案历史）+ 术语库（语义） |
| N2 检索对比 | 语义记忆（法规 / 现有技术）+ 经验记忆（相似案例） |
| N3 / N4 撰写 | 经验记忆（用户撰写风格画像）+ 模板库 + 案件记忆（历版） |
| N7 人审 | 案件记忆（版本对比） |

### 5. 写：固化流程

```jsx
node 产出 → 进工作记忆(state)
            │
        N6 校验 / N7 人审
            │ 通过
   ┌────────┴────────┐
固化进案件记忆(版本链)   抽取范式 → 经验记忆
```

### 6. 记忆 vs 知识库的区分

> **知识库**是公共、相对静态的领域知识（法规、范文、术语）；**记忆**是 agent 在交互中积累的、动态且个性化的内容（用户画像、案件历史、撰写经验）。两者都走向量检索，但生命周期、写入门控、隔离粒度完全不同——不要混在一个库里。
>

## 十、LLM Wiki：用户画像与长期记忆的结构化表示

<aside>
💡

**核心**：LLM Wiki 不是第五类记忆，而是长期记忆（用户画像 / 案件 / 经验）的一种**表示与维护方式**——把记忆从不可读的向量碎片，升级成人和模型都能读写、按实体分页、可交叉链接的 wiki 文档。本质是「编译后的记忆视图」。

</aside>

### 1. 为什么用 wiki 而不是纯向量记忆

| 维度 | 纯向量记忆 | LLM Wiki |
| --- | --- | --- |
| 可读性 | 黑箱碎片 | 人类可读 |
| 可编辑 | 难（只能重嵌入） | 律师 / 用户可直接改 |
| 去重一致 | 碎片重复、易冲突 | 合并消解后单一真相 |
| 组织 | 扁平 chunk | 按实体分页 + 交叉链接 |
| 召回 | 相似度命中片段 | 按实体 key 取整页（+ 分节） |

### 2. Wiki 的组织：按实体分页 + 交叉链接

- **用户 / 客户画像页**：撰写风格、术语偏好、好恶、历史反馈。
- **案件页（per case）**：技术方案摘要、权利要求布局决策、检索结论、OA 历史。
- **审查员页**：审查倾向、常见驳回理由。
- **技术领域页**：领域术语、常用现有技术、撰写惯例。
- **交叉链接**：案件页 ↔ 用户页 ↔ 领域页 ↔ 审查员页（类似 Notion 双链）。
- **层级**：索引页 + 详情页，避免单页膨胀。

### 3. 双层记忆：原始日志 + Wiki

```jsx
原始情景日志（append-only、向量、全证据）  ← 源
        │ 巩固 / 反思（consolidation）
Wiki（提炼、结构化、可编辑）              ← 编译产物
```

- 原始层不可改、可追溯；wiki 层高信号、可编辑。
- wiki 每条断言带 **provenance 回链**到原始来源（哪个案件 / 版本 / 对话）。

### 4. 写：巩固节点（consolidation / reflection）

- **触发**：任务结束 / 定期 / 人审通过后。
- **流程**：读现有 wiki 页 + 新证据 → 产出**增量 patch** 而非整页重写 → 去重、合并、冲突消解 → 更新。
- **门控**：只有 N6 校验 / N7 人审通过的信息才进 wiki。
- **更新策略**：结构化分节 + 定向 patch（与你的 AST / 分段思路一致：先定位分节再改）。

### 5. 读：按实体取整页注入

| Node | 注入的 wiki 页 |
| --- | --- |
| 改写层 / N1 | 用户画像页 + 当前案件页 |
| N2 检索对比 | 技术领域页 + 审查员页 |
| N3 / N4 撰写 | 用户画像页（撰写风格） |

页太大时 → 分节检索 / 索引页路由，而非整页塞进上下文。

### 6. 难点

- **写竞争**：并发更新同一页 → 加锁 / 串行巩固。
- **漂移与膨胀**：定期 compaction（压缩、归并过期断言）。
- **整页重写 vs 定向 patch**：优先 patch，降低「灾难性遗忘」。
- **信任**：人可改，但要留版本与 provenance。
- **与新证据冲突**：wiki 是「当前最佳信念」，遇强证据触发更新而非盲信旧页。

### 7. 落到你的工作空间

<aside>
🗂️

顺带一提：Notion 本身就是 wiki。案件页 / 用户画像页 / 审查员页可以直接用 Notion 数据库承载，agent 通过读写这些页面来维护 LLM Wiki——人和 agent 共用同一份记忆，天然可编辑、可追溯。

</aside>
